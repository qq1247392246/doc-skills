# DOC Skills - 面向文档编程

## 触发条件（B+C 混合）

### 方式 B：AI 自动识别

当用户输入**不包含** `/doc` 命令，但内容符合以下特征时：

**场景 1：需求讨论阶段**
- 描述新功能/需求/改进（如"我想做一个..."、"需要实现..."、"帮我加个..."、"能不能做..."、"想要..."）
- 讨论业务逻辑或技术方案
- 提到具体的功能点、模块、接口设计、重构、优化、改进、调整

**行为：**
- **必须**主动询问："检测到您可能在讨论新需求/改进，是否启动文档模式记录？"
- 用户确认后，**立即**按「创建新主题」流程执行
- 用户拒绝后，正常对话继续

**场景 2：需求完成阶段**
- 表达完成状态（如"已完成"、"已实现"、"做完了"、"写完了"、"搞定了"）
- 提到测试/验证通过（如"测试通过"、"验证OK"、"没问题了"）
- 询问下一步（如"接下来做什么"、"还有其他的吗"）

**行为：**
- **必须**主动询问："检测到实现完成，是否记录为文档并更新状态为「🚀 已实现」？"
- 用户确认后，执行状态更新并**立即更新 INDEX.md**
- 用户拒绝后，正常对话继续

### 方式 C：命令触发

用户输入以 `/doc` 开头时，直接激活技能，无需询问。

| 命令格式 | 行为 |
|----------|------|
| `/doc` | 询问主题名称，然后创建 |
| `/doc [主题名]` | 直接使用该主题名创建 |
| `/doc list` | 列出所有主题（自动同步索引） |
| `/doc open [主题]` | 继续已有主题 |
| `/doc done [主题]` | 标记主题为已实现 |
| `/doc del [主题]` | 删除主题 |
| `/doc sync` | 手动同步索引与文件系统 |

## 命令处理

### `/doc [主题名]`

**无主题名时：**
- 询问用户："请描述本次需求的主题名称"

**有主题名时，按顺序执行**：
1. 提取/确认主题名
2. 检查当前是否有未完成主题，有则询问是否切换
3. 创建目录：`doc/YYYY.MM.DD主题名/`
4. 从当前 skill 所在目录的 `templates/req.md` 复制模板创建 `主题名需求文档.md`
5. **立即更新 `doc/INDEX.md`**：追加新行，状态为「📝 草稿」
6. 告知用户："已创建主题「XXX」，状态：📝 草稿，索引已更新。请开始描述您的需求，我会帮您澄清并记录。"

### `/doc list`

**执行前强制检查**：
1. 扫描 `doc/` 目录下所有主题文件夹
2. 对比 INDEX.md 中的条目
3. 清理 INDEX.md 中已不存在的主题
4. 若有清理，告知用户："已同步索引，移除 N 个已删除的主题"

读取 `doc/INDEX.md`，以简化格式输出：
```
📚 文档列表：
  📝 主题A (草稿)
  ✅ 主题B (已确认)
  🚀 主题C (已实现)
```

### `/doc open [主题]`

1. 模糊匹配主题名（部分匹配即可）
2. 读取对应需求文档的「背景与目标」「核心需求」部分
3. 输出摘要："已切换到主题「XXX」(状态：📝)。当前进展：..."

### `/doc del [主题]`

1. 确认主题存在
2. 询问用户："确认删除主题「XXX」及其所有文档？"
3. 确认后删除文件夹并更新 INDEX.md

### `/doc done [主题]`

标记主题为「🚀 已实现」状态：
1. 若未指定主题，使用当前活跃主题
2. 确认主题存在且状态为「✅ 已确认」
3. 更新需求文档状态为「🚀 已实现」
4. **立即更新 INDEX.md**：状态列改为「🚀 已实现」，更新时间列
5. 告知用户："主题「XXX」已标记为已实现，索引已更新"

### `/doc sync`

手动触发索引对账：
1. 扫描 `doc/` 目录下所有主题文件夹
2. 对比 INDEX.md 中的条目
3. 清理不存在的主题，补充缺失的主题
4. 输出同步报告："已同步索引 - 移除 N 个，补充 M 个"

## 需求讨论流程（激活后自动执行）

### 1. 关键点回查

用户每次输入后：
- 识别描述中的关键词（功能名、模块名、数据实体等）
- 扫描 `doc/` 下所有文档的「核心需求」「业务逻辑」部分
- 若发现关联（相似功能、依赖模块、数据上下游）：
  - 静默读取关联文档的关键内容
  - 告知用户：「参考「XXX」文档中的「YYY」部分作为本次讨论依据」

### 2. 质疑循环

检查以下维度，发现任一缺失则批量提问：

| 检查项 | 质疑内容 |
|--------|----------|
| 合理性 | 这个需求是否合理？是否有更简单的替代方案？ |
| 明确性 | 「XXX」具体指什么？边界在哪里？ |
| 完整性 | 异常情况如何处理？前置条件是什么？ |
| 可实现性 | 现有系统是否支持？需要哪些改造？ |

**执行：**
- 汇总所有疑问，一次性向用户提问
- 用户回答后，重复检查直到无疑问
- 询问用户："需求已明确，是否确认并生成设计文档？"

### 3. 确认后处理

用户确认后，**按顺序执行**：
1. 将需求文档状态更新为「✅ 已确认」
2. 从当前 skill 所在目录的 `templates/design.md` 复制模板创建 `主题名设计文档.md`
3. 基于需求内容自动填充设计文档的「技术方案」部分（伪代码级）
4. **立即更新 INDEX.md**：状态列改为「✅ 已确认」，更新时间列
5. 告知用户："已生成设计文档，路径：... 索引已更新。可以开始实现了。"

### 4. 变动检测

每次用户输入时：
- 对比当前输入与已记录的需求内容
- 若发现实质差异（非措辞调整）：
  - 告知变更内容："检测到需求变更：原XX，现XX"
  - 询问："是否更新文档？"
  - 确认后更新需求文档并同步调整设计文档
  - **立即更新 INDEX.md**：更新时间列，告知用户"索引已同步"

## 状态流转

```
📝 草稿
  → 用户确认无疑问 → ✅ 已确认（生成设计文档）
  → 代码实现完成 + 编译通过 + 验收条件满足 → 🚀 已实现
```

**强制执行**：状态变更时，**必须立即**更新 INDEX.md 对应行的状态图标和更新时间，完成后告知用户："已更新索引：主题「XXX」状态变更为 [状态]"。

## 文档模板规范

### 需求文档结构
- 背景与目标（痛点/目标/优先级）
- 核心需求（功能逻辑/设计要点）
- 业务逻辑（主流程/异常流程/数据流向）
- 约束与边界（不做/限制）
- 依赖与前置条件

### 设计文档结构
- 技术方案（概述/关键设计/兼容性）
- 关键实现（伪代码/接口/数据结构）
- 文件变更清单
- 验收条件（功能/非功能）
- 风险与注意事项

## INDEX.md 维护

**强制性规则**：以下操作完成后，**必须立即**更新 `doc/INDEX.md`，不得遗漏：
- 新建主题（`/doc` 或方式 B 触发）
- 状态变更（草稿→已确认→已实现）
- 删除主题（`/doc del`）
- 需求文档或设计文档有实质性修改

自动维护以下格式的索引文件：

```markdown
# 📚 文档索引

## 统计
- 总计：N个主题
- 草稿：N | 已确认：N | 已实现：N

---

## 文档列表

| 主题 | 摘要 | 状态 | 标签 | 创建 | 更新 | 路径 |
|------|------|------|------|------|------|------|
| 主题A | 实时数据处理... | 📝 草稿 | 后端 | 03-03 | 03-03 | [📁](./...) |
```

**维护时机：**
- 新建主题：追加行
- 状态变更：更新状态列和时间列
- 删除主题：移除行并更新统计

## 输出风格

- 简洁直接，不冗余
- 使用 emoji 图标区分状态（📝/✅/🚀）
- 告知用户时明确文件路径，方便查阅
- 质疑问题批量汇总，不打断思路
