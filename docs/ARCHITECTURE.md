# DOC Skills 架构设计文档

> 本文档面向开发者/维护者，说明 Skills 的设计思想、工作原理和修改方法。

---

## 一、核心思想

### 1.1 面向文档编程（Document-Oriented Programming）

传统开发流程：
```
口头需求 → 直接写代码 → 代码即文档（实际没有文档）
```

DOC Skills 流程：
```
口头需求 → AI 质疑澄清 → 需求文档 → 设计文档 → 代码实现
                           ↓              ↓
                        可追溯        可验收
```

**核心理念**：
- 需求必须被**明确记录**才能进入开发
- AI 主动**质疑**而非被动接受
- 文档是**活**的，随讨论自动更新

### 1.2 解决的问题

| 问题 | 传统方式 | DOC Skills 方案 |
|------|---------|----------------|
| 需求模糊 | 口头传达，易遗忘 | AI 强制澄清，书面记录 |
| 上下文丢失 | 跨会话忘记之前讨论 | 文档持久化，随时回查 |
| 需求变更 | 口头变更，无记录 | 自动检测差异，更新文档 |
| 新人上手 | 需要长时间讲解 | 读文档即可理解系统 |

## 二、系统架构

### 2.1 触发机制（B+C 混合）

```
用户输入
    │
    ├── 包含 "/doc" 命令 ──────→ 直接激活（方式 C）
    │
    └── 自然语言描述需求 ─────→ AI 询问是否激活（方式 B）
                                    │
                                    ├── 用户确认 → 激活
                                    └── 用户拒绝 → 正常对话
```

**设计意图**：
- 方式 C：明确意图，高效直达
- 方式 B：防止遗漏，自动提醒

### 2.2 核心工作流

```
激活
  │
  ├── 方式1: 创建新主题 ────────┐
  │   - 提取主题名              │
  │   - 创建 doc/YYYY.MM.DD主题名/ 文件夹
  │   - 复制 templates/req.md    │
  │                             │
  ├── 方式2: 继续旧主题 ────────┤
  │   - /doc open 主题名        │
  │   - 加载已有文档            │
  │                             │
  └── 方式3: 列出所有主题 ──────┘
      - /doc list

激活后流程（自动执行）：
  │
  ├─→ 关键点回查 ──→ 扫描历史文档，发现关联
  │       │
  │       └── 告知用户参考依据
  │
  ├─→ 质疑循环 ────→ 检查需求完整性
  │       │
  │       ├── 有疑 ─→ 批量提问 → 循环
  │       └── 无疑 ─→ 询问确认
  │
  ├─→ 确认后处理 ──→ 生成设计文档
  │       │
  │       └── 复制 templates/design.md
  │
  └─→ 变动检测 ────→ 监控需求变更
          │
          ├── 有变动 ─→ 询问更新 → 更新文档
          └── 无变动 ─→ 继续讨论
```

### 2.3 文档状态机

```
         创建主题
            │
            ▼
    ┌───────────────┐
    │    📝 草稿     │◄─────────────┐
    │   (需求讨论中)  │              │
    └───────┬───────┘              │
            │ 用户确认              │
            ▼                       │
    ┌───────────────┐              │
    │   ✅ 已确认    │              │
    │ (需求明确，    │              │
    │  可进入开发)   │              │
    └───────┬───────┘              │
            │ 实现完成              │
            │ + 编译通过            │
            │ + 验收通过            │
            ▼                       │
    ┌───────────────┐              │
    │   🚀 已实现    │──────────────┘
    │   (开发完成)   │   需求变更
    └───────────────┘
```

## 三、模块职责

### 3.1 触发识别模块

**职责**：判断是否需要激活 Skills

**规则**：
- 命令触发：输入以 `/doc` 开头
- 语义触发：描述包含"想做一个"、"需要实现"、"功能"、"模块"等关键词

**修改方法**：
编辑 `skill.md` 中的"触发条件"部分，添加/删除关键词。

### 3.2 主题管理模块

**职责**：创建、切换、删除主题

**数据存储**：
- 文件夹：`doc/YYYY.MM.DD主题名/`
- 命名规则：日期前缀 + 主题名（去除非法字符）

**修改方法**：
编辑 `skill.md` 中的"命令处理"部分，修改路径或命名规则。

### 3.3 关键点回查模块

**职责**：发现当前讨论与历史文档的关联

**算法**：
1. 提取当前输入关键词（功能名、模块名、数据实体）
2. 扫描 `doc/` 下所有文档的「核心需求」「业务逻辑」
3. 匹配相似度 > 阈值则视为关联
4. 静默读取关联内容，告知用户

**修改方法**：
编辑 `skill.md` 中的"关键点回查"部分，调整匹配逻辑。

### 3.4 质疑循环模块

**职责**：确保需求完整、明确、可实现

**检查维度**：
| 维度 | 检查内容 | 缺失时行为 |
|------|---------|-----------|
| 合理性 | 需求是否合理？有替代方案？ | 建议更优方案 |
| 明确性 | 关键词定义？边界？ | 要求具体说明 |
| 完整性 | 异常处理？前置条件？ | 补充场景分析 |
| 可实现性 | 技术可行性？改造范围？ | 评估工作量 |

**修改方法**：
编辑 `skill.md` 中的"质疑循环"表格，添加/删除检查项。

### 3.5 文档生成模块

**职责**：创建和维护文档

**模板系统**：
- 需求模板：`templates/req.md`
- 设计模板：`templates/design.md`

**占位符替换**：
- `[主题名称]` → 实际主题名
- `YYYY-MM-DD` → 当前日期

**修改方法**：
- 修改模板内容：编辑 `templates/` 下的文件
- 修改占位符：编辑 `skill.md` 中的模板处理逻辑

### 3.6 索引维护模块

**职责**：维护 `doc/INDEX.md`

**格式**：
```markdown
| 主题 | 摘要 | 状态 | 标签 | 创建 | 更新 | 路径 |
```

**维护时机**：
- 新建主题：追加行
- 状态变更：更新状态列
- 删除主题：移除行

**修改方法**：
编辑 `skill.md` 中的"INDEX.md 维护"部分，调整表格列或格式。

## 四、文件结构详解

### 4.1 项目文件

| 文件 | 用途 | 修改场景 |
|------|------|---------|
| `skill.md` | AI 执行手册 | 调整工作流程、修改命令 |
| `templates/req.md` | 需求文档模板 | 调整需求文档结构 |
| `templates/design.md` | 设计文档模板 | 调整设计文档结构 |
| `docs/ARCHITECTURE.md` | 本文件 | 理解架构、指导修改 |
| `docs/spec.md` | 详细规范 | 参考实现细节 |

### 4.2 运行时生成的文件

```
doc/
├── INDEX.md                    # 自动维护的索引
└── 2025.03.03实时数据处理/      # 主题文件夹
    ├── 实时数据处理需求文档.md
    └── 实时数据处理设计文档.md
```

## 五、常见修改场景

### 5.1 添加新命令

**示例**：添加 `/doc search [关键词]` 搜索主题

1. 编辑 `skill.md`，在"命令处理"部分添加：
```markdown
### `/doc search [关键词]`

1. 扫描所有主题文件夹名称
2. 模糊匹配关键词
3. 返回匹配的主题列表
```

2. 更新 README.md 的命令表格

### 5.2 修改模板结构

**示例**：需求文档增加"用户故事"章节

1. 编辑 `templates/req.md`，添加：
```markdown
## 用户故事

作为 [角色]，我想要 [功能]，以便 [价值]。
```

2. 编辑 `skill.md`，更新"需求文档结构"说明

### 5.3 调整质疑维度

**示例**：增加"安全性检查"

1. 编辑 `skill.md` 的"质疑循环"表格：
```markdown
| 安全性 | 是否有敏感数据？权限控制？ | 补充安全方案 |
```

### 5.4 修改状态流转

**示例**：增加"已评审"状态

1. 编辑 `skill.md` 的"状态流转"：
```
草稿 → 已确认 → 已评审 → 已实现
```

2. 修改状态变更触发条件

## 六、设计原则

### 6.1 简洁性
- 每个模块职责单一
- 不引入不必要的依赖
- 模板保持可阅读性

### 6.2 可扩展性
- 模板系统易于修改
- 检查维度可插拔
- 命令系统可扩展

### 6.3 用户友好
- 不强迫，只提醒
- 批量提问，减少打断
- 明确告知文件路径

---

## 七、故障排查

### 7.1 AI 不识别命令

- 检查 `skill.md` 是否正确复制到配置目录
- 检查文件编码是否为 UTF-8
- 重启 Claude Code / Codex

### 7.2 模板未正确复制

- 检查 `templates/` 文件夹是否和 `skill.md` 在同一目录
- 检查文件权限

### 7.3 文档未生成

- 检查当前工作目录是否有写权限
- 检查 `doc/` 文件夹是否被正确创建
