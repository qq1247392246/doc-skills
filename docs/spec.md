# DOC Skills 规范文档

> 面向文档编程辅助工具 — 让需求讨论沉淀为可追溯的结构化文档

---

## 一、定位与职责

本 Skills 的核心职责：
1. **需求澄清**：质疑用户需求的合理性、明确性与实现细节，直到无疑问
2. **文档生成**：将讨论结果自动写入结构化文档（需求文档 + 设计文档）
3. **文档维护**：检测变动并自动更新文档，保持文档与讨论同步
4. **关键点回查**：在需求讨论中自动识别与已有文档的关联，静默查询并告知用户参考依据

---

## 二、触发方式（B+C 混合）

| 方式 | 说明 |
|------|------|
| **主动触发** | 用户执行 `/doc` 命令，明确启动文档模式 |
| **AI 识别提示** | AI 检测到用户正在描述需求/功能时，主动询问"是否启动文档模式？" |
| **自动延续** | 同一主题启动后，后续输入自动延续，无需重复触发 |

> AI 常驻文档编程模式，无需退出命令。

---

## 三、命令全集

| 命令 | 功能 |
|------|------|
| `/doc` | 启动新主题的文档模式 |
| `/doc list` | 展示所有主题及状态（简化：主题 + 状态） |
| `/doc open [主题]` | 继续旧主题，AI 主动摘要当前进展 |
| `/doc del [主题]` | 删除整个主题文件夹（含需求文档 + 设计文档） |

## 四、核心工作流

### 4.1 启动新主题

```
用户触发 /doc 或 AI 识别到需求讨论
  → AI 自动提取主题名，告知用户："本次将以「XXX」作为主题保存，如不满意请告诉我，您想要的主题名"
  → 用户确认或修改主题名
  → 创建文件夹：doc/YYYY.MM.DD主题名/
  → 创建需求文档（草稿状态）
```

### 4.2 关键点回查

```
用户每次输入
  → AI 识别当前讨论内容与已有文档的潜在关联
  → 若发现关联：直接查询关联文档的关键信息
  → 告知用户：「参考「XXX」文档中的「YYY」部分作为本次质疑/讨论依据」
  → 基于回查结果继续后续流程
```

> 回查不询问用户，直接执行并告知参考依据，避免打断思路。

### 4.3 需求质疑循环

```
用户每次输入（结合回查结果）
  → AI 检查：需求是否合理？是否明确？实现细节是否完整？
  → 若有疑问：汇总所有问题，批量向用户提问
  → 用户逐一回答
  → 重复上述过程，直到 AI 无疑问
  → AI 询问用户："需求已明确，是否确认？"
  → 用户确认 → 需求文档状态变更为「已确认」
             → 自动生成设计文档，并通知用户
```

### 4.4 单主题锁定

```
当前有未完成主题时，用户试图开启新主题
  → AI 询问："当前主题「XXX」尚未完成，是否切换到新主题？"
  → 用户确认后切换
```

### 4.5 文档自动更新

```
用户输入内容与现有文档存在差异
  → AI 检测到变动，告知用户："检测到需求变更：XXX，是否更新文档？"
  → 用户确认 → 更新文档 → 通知用户"文档已更新"
```

### 4.6 状态自动流转

```
草稿
  → 用户确认需求无疑问后 → 已确认（同步生成设计文档）
  → 代码实现完成 + 编译通过 + 验收条件全部满足 → 已实现
```

## 五、文档结构规范

### 5.1 目录结构

```
doc/
├── INDEX.md                          # 全局索引，自动维护
└── YYYY.MM.DD主题名/                  # 讨论主题文件夹
    ├── 主题名需求文档.md
    └── 主题名设计文档.md
```

### 5.2 文件命名规则

| 类型 | 命名格式 | 示例 |
|------|----------|------|
| 文件夹 | `YYYY.MM.DD主题名` | `2026.02.14实时数据逻辑变更` |
| 需求文档 | `主题名需求文档.md` | `实时数据逻辑变更需求文档.md` |
| 设计文档 | `主题名设计文档.md` | `实时数据逻辑变更设计文档.md` |

### 5.3 文档模板

参考：`templates/req.md` 与 `templates/design.md`

| 文档 | 核心内容 |
|------|----------|
| 需求文档 | 背景目标 / 核心需求 / 业务逻辑 / 约束边界 / 依赖条件 |
| 设计文档 | 技术方案 / 关键实现 / 接口设计 / 文件变更 / 验收条件 / 风险 |

### 5.4 版本策略

直接覆盖，不保留历史版本。

## 六、INDEX.md 维护规则

### 6.1 格式

```markdown
# 📚 文档索引

## 统计
- 总计：N个主题
- 草稿：N | 已确认：N | 已实现：N

---

## 文档列表

| 主题 | 摘要 | 状态 | 标签 | 创建 | 更新 | 路径 |
|------|------|------|------|------|------|------|

---

## 状态说明
- 📝 草稿：需求讨论中
- ✅ 已确认：需求明确，可进入开发
- 🚀 已实现：开发完成
```

### 6.2 自动维护时机

| 事件 | 动作 |
|------|------|
| 新建主题 | 追加一行记录 |
| 状态变更 | 更新对应行状态 |
| 文档更新 | 更新「最后更新」时间 |
| 删除主题 | 移除对应行，更新统计 |

---

## 七、行为约束

1. **一次一主题**：当前主题未完成时，切换前必须询问用户确认
2. **质疑优先**：每次用户输入都检查需求完整性，有疑问必须提问
3. **确认后才生成设计文档**：需求未确认前不生成设计文档
4. **变更必确认**：检测到需求变动时，必须经用户确认才更新文档
5. **只保留最终结论**：质疑过程的问答不写入文档，只保留最终完整需求
6. **标签自由生成**：AI 根据内容自动生成标签，无预设列表
